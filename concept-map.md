# concept-map

사용자가 `/concept-map [개념]`을 입력하거나, 특정 개념의 구조·위치·경쟁 관계·시장 맥락에 대한 설명을 요청하면 아래 절차로 실행한다.

## 개체 분류 규칙

설명 대상을 먼저 아래 6가지 중 하나로 분류하고, **출력 시작 시 반드시 분류 결과를 표시**한다.

| 분류 | 기준 |
|------|------|
| 기업(Company) | 법인 조직. 제품·서비스를 만드는 주체 |
| 제품(Product) | 기업이 출시한 소프트웨어·하드웨어·서비스 |
| 기술(Technology) | 특정 방법론·알고리즘·아키텍처 패턴 |
| 시장(Market) | 수요·공급이 형성된 산업 카테고리 |
| 표준/프로토콜(Standard) | 업계 합의로 만들어진 규격·인터페이스 |
| 플랫폼(Platform) | 다수의 공급자·소비자를 연결하는 생태계 기반 |

분류 예시:
- Rossum → **기업**
- Rossum Aurora → **제품**
- T-LLM → **기술**
- IDP → **시장**
- REST API → **표준/프로토콜**
- AWS → **플랫폼**

## 실행 규칙

- 설명 전체에서 **기술 / 기업 / 제품 / 시장**을 명확히 구분해서 서술한다.
- 분류가 필요한 항목(3·4·5번)은 **MECE 원칙**에 따라 겹치지 않고(ME) 빠짐없이(CE) 구성한다.
- 단순 정의가 아니라, 개념이 속한 **시장·기술 지형도** 안에서 위치와 맥락을 설명한다.

## 출력 구조

### 0. 개체 분류
> [설명 대상] → **[분류]**
> 한 줄 요약: [이게 무엇인지 한 문장]

### 1. 상위 개념
이 개념이 속한 더 큰 범주. 기술 계층인지, 산업 분류인지, 시장 카테고리인지 명시.

### 2. 존재 목적
왜 이 개념·제품·기술이 등장했는가? 어떤 문제나 공백을 해결하기 위해 존재하는가?

### 3. 동일 층위 비교 (MECE)
같은 레벨의 유사 개념·제품·기업. 겹치지 않게, 빠짐없이 각각 2~3줄로 비교.

### 4. 대체재 / 경쟁재 (MECE)
직접 경쟁(같은 방식)과 간접 대체(다른 방식으로 같은 목적 달성)를 구분해서 설명.

### 5. 하위 구성요소 (MECE)
겹치지 않고 빠짐없이 모듈·기능·레이어·역할로 분류.

### 6. 실제 사용 예시
실제 기업이나 제품에서 쓰이는 구체적 사례 2~3개.

### 7. 트리 구조
```
[최상위 개념]
├── [상위 개념]
│   ├── [설명할 개념]  ← 여기
│   │   ├── [하위 요소 1]
│   │   └── [하위 요소 2]
│   └── [동일 층위 비교 개념]
└── [다른 상위 개념]
```

### 8. 비유
기술을 모르는 사람도 이해할 수 있는 일상적 비유.
