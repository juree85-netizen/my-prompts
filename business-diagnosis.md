# business-diagnosis

## 핵심 원칙

> **BMC는 문제 위치를 찾는 도구이고, 맥킨지 7단계는 원인 제거 및 실행 우선순위를 설계하는 도구이다.**  
> **BMC 진단 결과(Root Cause Block)는 반드시 McKinsey Step 2(Issue Tree)의 최상위 노드로 연결한다.**

---

## BMC 9개 블록

| 블록 | 코드 | 핵심 질문 |
|------|------|-----------|
| 고객 세그먼트 | CS | 누구를 위한 사업인가? |
| 가치 제안 | VP | 어떤 문제를 해결해주는가? |
| 채널 | CH | 어떻게 고객에게 전달하는가? |
| 고객 관계 | CR | 고객과 어떤 관계를 맺는가? |
| 수익 흐름 | R$ | 무엇으로 돈을 버는가? |
| 핵심 자원 | KR | 무엇이 있어야 작동하는가? |
| 핵심 활동 | KA | 무엇을 해야 하는가? |
| 핵심 파트너십 | KP | 누구와 함께해야 하는가? |
| 비용 구조 | C$ | 무엇에 돈이 드는가? |

---

## BMC 진단 규칙 (Root Cause Block 판단)

> **단순 결과(Symptom)는 원인 블록으로 판단하지 않는다.**  
> 문제를 가장 먼저 발생시키는 블록을 **Root Cause Block**으로 지정한다.

| 증상 | 잘못된 판단 | 올바른 Root Cause Block |
|------|------------|------------------------|
| 매출 감소 | R$ | VP / CH / CR 우선 검토 |
| 고객 이탈 증가 | CR | VP(가치 부족) 또는 CH(접점 문제) |
| 신규 고객 확보 실패 | CS | CH(채널 부재) 또는 VP(소구력 약화) |
| 원가 상승 | C$ | C$ (이 경우는 직접 원인) |
| 파트너 이탈 | KP | VP(파트너에게 돌아가는 가치 부족) 또는 R$(수익 배분 구조) |
| 운영 비효율 | KA | KR(자원 부족) 또는 KA(프로세스 설계 오류) |

**Root Cause 판단 순서**:
1. 증상의 "바로 앞 단계" 블록이 무엇인지 추적
2. 복수 블록이 동시에 의심될 경우 → "더 먼저 발생한 것"이 Root Cause
3. R$·C$는 결과 블록인 경우가 많음 — 원인 블록을 먼저 탐색

## 문제 유형 분류 기준

| 유형 | 정의 | 대표 증상 |
|------|------|-----------|
| **Growth** | 성장 엔진이 멈춘 문제 | 매출 정체, 신규 고객 확보 실패, 시장점유율 하락 |
| **Profitability** | 수익성 구조 문제 | 매출 증가에도 이익 감소, 원가율 상승, 마진 압박 |
| **Operation** | 실행·운영 효율 문제 | 프로세스 병목, 납기 지연, 품질 저하, 인력 비효율 |
| **Organization** | 조직·인재 문제 | 핵심인력 이탈, 사일로, 의사결정 지연 |
| **Strategy** | 방향성·포지셔닝 문제 | 시장 변화 대응 실패, 경쟁 우위 소멸, 비즈니스 모델 노후화 |

## Issue Tree 유형별 템플릿

### Growth 문제
```
매출 성장 정체
├── 고객 수 감소
│   ├── 신규 유입 감소 (CH 문제)
│   └── 이탈률 증가 (VP / CR 문제)
├── 객단가 하락
│   ├── 가격 경쟁력 약화
│   └── 업셀·크로스셀 실패
└── 구매 빈도 감소
    ├── 재구매 동기 부족 (VP 문제)
    └── 경쟁사 전환
```

### Profitability 문제
```
수익성 악화
├── 매출총이익 하락
│   ├── 원가 상승 (C$ 문제)
│   └── 가격 하락 압력 (VP 약화)
├── 판관비 비효율
│   ├── 인건비 증가
│   └── 마케팅 ROI 하락 (CH 문제)
└── 비반복 매출 비중 과다 (R$ 구조 문제)
```

### Operation 문제
```
운영 비효율
├── 프로세스 병목
│   ├── 자원 부족 (KR 문제)
│   └── 활동 설계 오류 (KA 문제)
├── 파트너 의존 리스크
│   └── 단일 파트너 집중 (KP 문제)
└── 품질 저하
    ├── 핵심 역량 부재
    └── 실행 인력 부족
```

### Strategy 문제
```
전략적 경쟁력 약화
├── 가치 제안 노후화 (VP 문제)
│   ├── 고객 니즈 변화 미반영
│   └── 경쟁사 VP 우위 확보
├── 고객 세그먼트 변화 대응 실패 (CS 문제)
└── 수익 모델 한계 (R$ 문제)
    ├── 단일 수익원 의존
    └── 구독·반복 수익 부재
```

## 출력 구조

### 0. 먼저 보는 프레임워크: BMC란?

BMC(Business Model Canvas)는 사업을 9개의 블록으로 나누어 보는 사업모델 진단 도구이다.

이 스킬에서 BMC는 "사업 전체를 설명하는 도구"라기보다, **문제가 사업모델의 어느 위치에서 시작됐는지 찾는 진단 지도**로 사용한다.

9개 블록은 다음과 같다.

| 블록 | 핵심 질문 |
|------|-----------|
| CS (Customer Segments) | 누구를 위한 사업인가 |
| VP (Value Proposition) | 어떤 가치를 제공하는가 |
| CH (Channels) | 고객에게 어떻게 전달하는가 |
| CR (Customer Relationships) | 고객과 어떤 관계를 맺는가 |
| R$ (Revenue Streams) | 무엇으로 돈을 버는가 |
| KR (Key Resources) | 필요한 핵심 자원은 무엇인가 |
| KA (Key Activities) | 반드시 수행해야 하는 핵심 활동은 무엇인가 |
| KP (Key Partnerships) | 누구와 협력해야 하는가 |
| C$ (Cost Structure) | 어떤 비용이 발생하는가 |

이후 분석에서는 증상 자체가 아니라, 문제가 처음 발생한 위치인 **Root Cause Block**을 찾아 McKinsey Issue Tree의 출발점으로 연결한다.

---

### 1. 사업 프로파일 + 문제 유형 분류

> **[사업 유형]**: B2B / B2C / 플랫폼 / 내부 서비스  
> **[문제 유형]**: Growth / Profitability / Operation / Organization / Strategy  
> **[선택 이유]**: [왜 이 유형으로 분류했는지 1~2줄]

---

### Phase 1 — BMC Root Cause 진단

**Root Cause Block** (문제 발생 기점)
- [블록명]: [이 블록이 왜 원인인지 — 증상이 아닌 발생 메커니즘으로 설명]

**연쇄 영향 블록**
- [블록명] ← [Root Cause Block]에서 어떻게 이어졌는지

**BMC 진단 요약**
```
[CS] [VP] [CH]
     [CR] [R$]
[KR] [KA] [KP]
          [C$]
← 🔴 Root Cause Block / 🟡 연쇄 영향 블록
```

> **Root Cause → Issue Tree 연결**:  
> [Root Cause Block]에서 발생한 [문제]가 McKinsey Step 2의 최상위 노드가 된다.

---

### Phase 2 — 맥킨지 7단계 문제해결

#### Step 1. 문제 정의 (Problem Definition)
- **문제 진술**: [SMART — 구체적·측정가능·관련성있게]
- **문제가 아닌 것**: [범위 명시적 제외]
- **성공 기준**: [해결됐다고 볼 수 있는 상태]

#### Step 2. 문제 구조화 (Issue Tree)
> 위 유형별 템플릿 기반으로 생성. **최상위 노드 = Phase 1의 Root Cause Block**

```
[Root Cause Block에서 도출된 핵심 문제]
├── [원인 가설 A — MECE]
│   ├── [하위 이슈 A-1]
│   └── [하위 이슈 A-2]
├── [원인 가설 B]
│   └── [하위 이슈 B-1]
└── [원인 가설 C]
    └── [하위 이슈 C-1]
```

#### Step 3. 우선순위 설정 (Priority Score)

> **Priority Score = Impact × Feasibility** (각 1~5점, 만점 25점)

| 이슈 | Impact (1~5) | Feasibility (1~5) | Score | 순위 |
|------|-------------|-------------------|-------|------|
| [이슈 A] | 5 | 5 | 25 | 1 |
| [이슈 B] | 4 | 4 | 16 | 2 |
| [이슈 C] | 3 | 2 | 6 | 3 |

#### Step 4. 분석 계획 (Work Plan)
| 이슈 | 필요한 데이터·분석 | 확인 방법 |
|------|------------------|----------|
| [이슈 A] | [무엇을 봐야 하는가] | [어디서/어떻게] |

#### Step 5. 핵심 발견사항 (Key Findings)
- **Finding 1**: [데이터·논리 기반 — "~이기 때문에 ~이다" 구조로 작성]
- **Finding 2**: ...
- **Finding 3**: ...

#### Step 6. 종합 — So What?

> **작성 규칙**: 1문장, 20~40자  
> 형식: "문제의 핵심은 ○○이며, 가장 효과적인 해결책은 ○○이다."

> [So What 한 문장]

#### Step 7. 권고안 + Elevator Pitch

| 순위 | 액션 | Impact | Feasibility | Score | 실행 주체 | 시점 |
|------|------|--------|------------|-------|----------|------|
| 1 | [즉시 액션] | 5 | 5 | 25 | [누가] | 즉시 |
| 2 | [단기 과제] | | | | | 1~3개월 |
| 3 | [중기 과제] | | | | | 3~6개월 |

**Elevator Pitch (30초, 4문장)**
> 1. **현재 문제**: [지금 무슨 일이 일어나고 있는가]  
> 2. **근본 원인**: [왜 그 문제가 생겼는가 — Root Cause Block 기반]  
> 3. **해결책**: [무엇을 해야 하는가]  
> 4. **기대 효과**: [해결하면 어떻게 달라지는가]
